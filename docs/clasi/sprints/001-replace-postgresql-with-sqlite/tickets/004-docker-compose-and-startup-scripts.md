# Ticket 004: Docker Compose and Startup Scripts

## Description

Update the Docker infrastructure to remove the Postgres `db` service and
replace it with a SQLite volume mount on the `server` service. Also fix the
container startup scripts to remove the TCP readiness probe and update the
migration command.

**Important context on compose file location:** The project uses rundbat to
generate Docker Compose files. The generated files land in `config/files/`
(currently empty per the project scan). The compose file paths in
`config/rundbat.yaml` point to `docker/docker-compose.yml` and
`docker/docker-compose.prod.yml`. Before editing, verify whether these
files exist (they may not have been generated yet, or they may be at
different paths). If the compose files are managed by rundbat, the correct
workflow is to edit `config/rundbat.yaml` and re-run `rundbat generate`.
If the compose files are hand-maintained at the paths in `rundbat.yaml`,
edit them directly.

## Acceptance Criteria

- [x] The `db` Postgres service is absent from both compose files (dev and prod).
- [x] The `server` service in both compose files mounts a named volume for
      SQLite: e.g., `- sqlite-data:/app/data`.
- [x] A top-level `volumes: sqlite-data:` entry is present in both compose files.
- [x] `DATABASE_URL` in the server service environment points to the mounted
      path: `file:/app/data/dev.db` (dev) / `file:/app/data/prod.db` (prod).
- [x] `docker/dev-server-start.sh` no longer contains `npx wait-on tcp:db:5432`.
- [x] `docker/dev-server-start.sh` migration command is
      `npx drizzle-kit migrate` (not `prisma migrate`).
- [x] `docker/wait-for-db.sh` is deleted.
- [x] `package.json` root-level `dev:local:db` script is removed or replaced
      (it currently starts a Postgres container; it has no SQLite equivalent).
- [x] `npm run dev:docker` starts the application with only `server` and
      `client` containers.

## Implementation Plan

### Step 1: Locate compose files

Check whether `docker/docker-compose.yml` and `docker/docker-compose.prod.yml`
exist. If yes, edit them directly. If the compose files are generated by
rundbat, run `rundbat generate` after editing `config/rundbat.yaml` and verify
the output. The `container_template` and `database_template` fields in
`rundbat.yaml` are Postgres naming conventions — remove or neutralize them.

### Step 2: Edit compose files

For each compose file, apply these changes:

1. Remove the `db` service block entirely.
2. Remove any `depends_on: [db]` entries from the `server` service.
3. Add a volume mount to the `server` service:
   ```yaml
   volumes:
     - sqlite-data:/app/data
   ```
4. Add the top-level volume declaration:
   ```yaml
   volumes:
     sqlite-data:
   ```
5. Set `DATABASE_URL` in the server service environment:
   ```yaml
   environment:
     DATABASE_URL: file:/app/data/dev.db   # or prod.db for prod
   ```
   (Or ensure it flows from the `config/*/public.env` which already has
   `DATABASE_URL=file:./data/dev.db`.)

### Step 3: Edit `docker/dev-server-start.sh`

Remove the `npx wait-on tcp:db:5432 --timeout 30000` line.
Replace the migration command block with:
```sh
echo "Running migrations..."
npx drizzle-kit migrate
```
Remove the Prisma fallback entirely.

### Step 4: Delete `docker/wait-for-db.sh`

```bash
rm docker/wait-for-db.sh
```

### Step 5: Update root `package.json`

Remove or comment the `dev:local:db` script (it runs a Postgres container).
Update `dev:local:server` to remove the `./docker/wait-for-db.sh` call:
```json
"dev:local:server": "set -a && . ./.env && set +a && cd server && npm run db:migrate && npm run dev"
```
(The `wait-for-db.sh` call is no longer needed.)

### Files to Modify or Delete

- `docker/docker-compose.yml` (if it exists) — remove db service, add volume
- `docker/docker-compose.prod.yml` (if it exists) — remove db service, add volume
- `config/rundbat.yaml` — remove `container_template`, `database_template`
  (if Postgres-specific); verify deploy workflow
- `docker/dev-server-start.sh` — remove TCP wait, fix migration command
- `docker/wait-for-db.sh` — delete
- `package.json` (root) — update `dev:local:db`, `dev:local:server` scripts

### Testing Plan

- `npm run dev:docker` starts without error; `docker compose ps` shows two
  services (server, client).
- `GET /api/health` returns 200 inside the Docker environment.
- Stop and restart the server container; verify data persists (volume mount
  is working).
